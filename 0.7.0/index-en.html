Abstract

Lightweight ontology for data spaces. It aligns IDSA and BIGOWL, focusing on the strong semantic description of Data Assets and Data Apps through Data Profiles, which include technical characteristics (DCAT 3) and granular quality metrics.

Introduction back to ToC

EDAAnOWL provides a compact yet expressive vocabulary to describe the operational semantics of data spaces and the analytical artefacts that flow through them. While the abstract motivates the need, this introduction clarifies scope, design choices, and typical usage. Scope. EDAAnOWL focuses on (i) participants and roles, (ii) data assets and their profiles, (iii) offers and usage policies, and (iv) analytical workflows and applications. It aligns with the International Data Spaces (IDSA) information model for contractual and policy notions, and connects to the BIGOWL family to describe algorithms, problems, and workflows—linking inputs/outputs through ports and data profiles. The ontology encourages the use of curated external concept schemes (AGROVOC, EU Data Theme, DQV) to ensure consistent and interoperable annotations. Reuse over reinvention. EDAAnOWL reuses widely adopted vocabularies: DCAT for metadata, SOSA/SSN for observations, OWL-Time for time, ODRL/IDSA for policies, DQV for quality and PROV-O for provenance. These alignments are expressed via lightweight bridges so existing assets can be integrated without duplication. Modular structure. The ontology distinguishes a stable core from project-specific modules. Reference vocabularies previously held locally have been externalized to international normative standards (FAO, EU, OGC), enabling global interoperability without local maintenance silos. Versioning and persistence. Each release has a persistent, versioned IRI. This enables unambiguous citation of a concrete release (“This version”) while preserving a canonical entry point (“Latest version”) to facilitate discovery and link persistence. Typical usage. - Describe a dataset (DataAsset): 1. Create a `dcat:Distribution` (or `:DataRepresentation`) and link it to the DataAsset via `ids:representation`. 2. Link the profile to the Distribution using `edaan:conformsToProfile`. - Define the profile: - Classify its type (e.g., `edaan:declaresDataClass bigdat:TabularDataSet`). - Define its technical properties (e.g., `dcat:temporalResolution "P1D"^^xsd:duration`, `edaan:hasCRS `). - List its metrics using `edaan:hasMetric`, describing each `edaan:Metric` with `:metricType` (aligned with `dqv:Metric`), `:metricValue` (the measured value), and `:appliesToFeature` (for column-level metrics). - Publish a DataApp: Declare the profiles it requires (`edaan:requiresProfile`) and produces (`edaan:producesProfile`), and its performance metrics with `:hasPerformanceMetric`. - Enable Matchmaking: Allow a broker to find assets that semantically match the quality and technical requirements of an application. - Validate Instances: Use the provided SHACL shapes, including integrated support for the IDSA Information Model (`idsa-shapes.ttl`) and EDAAnOWL specific rules (`edaan-shapes.ttl`). Non-goals. EDAAnOWL is not a monolithic domain ontology. It provides the semantic “connective tissue” required to publish, discover, and compose data and analytics across domains, delegating domain-specific semantics to external authoritative vocabularies.

EDAAnOWL: Description back to ToC

Ontology for data spaces that bridges the IDSA Information Model with the BIGOWL analytical family. Its core function is to describe `DataAssets` (assets) and `DataApps` (services) using `DataProfiles` (profiles): semantic models that detail technical specifications (like DCAT 3 resolutions, CRS) and granular quality metrics (like completeness or uniqueness per feature). This enables robust, automated discovery and compatibility checking (matchmaking) for data and services.
