Abstract

EDAAnOWL is a lightweight, modular ontology for describing the core semantics of data spaces—covering actors, roles, assets, offers/policies, processes, and interactions—so participants can publish, discover, and govern data under shared rules. It aligns with the International Data Spaces (IDSA) information model and connects with the BIGOWL family to describe analytical workflows, algorithms, problems, datasets, and input/output profiles. The ontology reuses well-established vocabularies (e.g., DCAT for cataloguing, SOSA/SSN for observations, OWL-Time for temporal descriptions) and provides curated concept schemes (sector, datatypes, observed properties) to enable consistent cross-domain annotation without imposing a single domain model. EDAAnOWL is versioned, import-friendly, and designed for extension through project-specific modules while preserving a stable core. It supports asset description, policy expression, provenance and lifecycle hooks, and SHACL-based validation to integrate heterogeneous resources and make data-sharing agreements and analytical pipelines transparent and machine-processable.

Introduction back to ToC

EDAAnOWL provides a compact yet expressive vocabulary to describe the operational semantics of data spaces and the analytical artefacts that flow through them. While the abstract motivates the need, this introduction clarifies scope, design choices, and typical usage. Scope. EDAAnOWL focuses on (i) participants and roles, (ii) data assets and their profiles, (iii) offers and usage policies, and (iv) analytical workflows and applications. It aligns with the International Data Spaces (IDSA) information model for contractual and policy notions, and connects to the BIGOWL family to describe algorithms, problems, and workflows—linking inputs/outputs through ports and data profiles. The ontology ships with curated concept schemes (sector, datatypes, observed properties, domain terms) to encourage consistent annotations without locking users into a single domain model. Reuse over reinvention. EDAAnOWL reuses widely adopted vocabularies wherever possible: DCAT for cataloguing and dataset metadata, SOSA/SSN for observations and features of interest, OWL-Time for temporal descriptions, ODRL and IDSA Core for policy/contract semantics, and PROV-O for provenance. These alignments are expressed via imports and lightweight bridges so existing assets can be described without duplication. Modular structure. The ontology distinguishes a stable core from project-specific modules and reference vocabularies. The core defines the main classes and properties (e.g., Actor, Role, DataAsset, DataApp, DataProfile, Offer/Policy, Workflow, InputPort/OutputPort). Reference vocabularies are versioned and replaceable, letting communities specialise terms while keeping interoperability at the interfaces. Versioning and persistence. Each release has a persistent, versioned IRI and a non-versioned namespace that denotes the latest version. This enables unambiguous citation of a concrete release (“This version”) while preserving a canonical entry point (“Latest version”), typically redirected to the documentation of the rolling latest/ build. Typical usage. • Describe a dataset (DCAT) and attach a DataProfile with schema/datatype constraints and observed properties. • Publish a DataApp or workflow, declaring its input/output ports and the expected/produced profiles (BIGOWL). • Express offers and usage policies with ODRL/IDSA, linking them to assets or services. • Record provenance of executions and derivations (PROV-O). • Validate instances with SHACL shapes distributed alongside the ontology. Non-goals. EDAAnOWL is not a monolithic domain ontology nor a replacement for sector models. It provides the semantic “connective tissue” required to publish, discover, govern, and compose data and analytics across domains, encouraging extension through clean interfaces rather than centralised control.
